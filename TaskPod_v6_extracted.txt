1. Product Overview
TaskPod is a lightweight Progressive Web App for small computer repair and service businesses. Version 6.0 is purpose-built for a 2-business deployment — the TaskPod operator (Superadmin) personally knows both business owners, collects payment offline, and activates accounts from the dashboard.
The Superadmin creates business accounts, sets up owner credentials, and can activate or suspend any account. Each business has one Owner who creates tasks and manages field Staff. Staff receive push notifications, update task status, and add remarks.

1.3 What Changed from v5.0
Major simplification: v6.0 removes the entire payment submission/approval system. Since the Superadmin personally collects offline payment from 2 known clients, the only actions needed are Activate and Suspend. No payment forms, no payment proof, no approval queue.
Business creation is Superadmin-only: No self-registration (/setup removed). Superadmin creates business + owner account and shares credentials via WhatsApp.
Per-business task numbering: Task #1001, #1002... is now scoped per business, not globally shared.
Kanban scope fix: Unassigned and In Progress columns show ALL tasks regardless of age. Only Completed is scoped to last 30 days.

5.2 End-to-End Flows
Owner Flow
Owner receives customer call → opens TaskPod → taps FAB (+)
Quick capture: enters Customer Name + Phone → taps "Save Draft"
Optionally fills Address, Service Type, assigns staff, marks Urgent, sets Due Date
Task saved with per-business auto-incremented number, appears on Kanban via Realtime
Owner assigns from card or detail → status = in_progress → push fires to staff
Staff Flow
Receives push: "New task assigned — Task #1005" (no customer PII in push body)
Taps notification → Task Detail opens directly → sees full customer details
Reads details, calls customer (tap-to-call)
Adds remark → taps "Mark as Completed" → confirmation → confirms
Owner receives push; task moves to Completed column via Realtime
Reassignment Flow
Owner reassigns from A to B → task_events row (Postgres trigger) → push to both A and B
Deactivating a staff member auto-moves all their tasks to status = unassigned; owner receives summary notification: "X tasks unassigned — [Staff Name] was deactivated"
Task Reopening Flow
Owner taps "Reopen Task" on completed task → confirmation bottom sheet
status → unassigned, completed_at → NULL, assigned_to → NULL
task_events row inserted; system remark auto-added
Task reappears in Unassigned column on Kanban
Draft Reminder Flow
Edge Function runs every 4 hours
For each business: count tasks WHERE is_draft = true AND created_at < now() - interval '4 hours'
If count > 0: push notification to owner — "You have X incomplete drafts"
Max 1 draft reminder per business per 8 hours (debounced)
Optimistic Concurrency Flow
Every task UPDATE includes WHERE updated_at = [last_known_updated_at]
If 0 rows affected: the task was modified by another user since it was loaded
Frontend shows conflict snackbar: "This task was updated by someone else. Refresh to see changes."
User taps refresh → re-fetches task → can retry their action

Enforcement trigger:
SQL
CREATE OR REPLACE FUNCTION enforce_max_businesses()
RETURNS TRIGGER AS $$
BEGIN
IF (SELECT COUNT(*) FROM businesses) >= 2 THEN
RAISE EXCEPTION 'Maximum of 2 businesses allowed on this platform';
END IF;
RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_max_businesses
BEFORE INSERT ON businesses
FOR EACH ROW EXECUTE FUNCTION enforce_max_businesses();

Superadmin seeding: The Superadmin user is seeded via a one-time SQL script run against the Supabase dashboard (INSERT into auth.users + users table with role = superadmin, must_change_password = false). There is no self-registration for Superadmin.
Owner creation: Superadmin creates the Owner via /sa/businesses/new. Edge Function creates auth.users entry + users row + businesses row atomically.
Phone uniqueness: Phone is NOT globally unique — same person could be owner of one business and have their phone appear as customer in another. However, within a business, staff phone numbers must be unique. This is enforced at the application level during staff creation, not via a DB constraint on users.phone.

Per-business task number trigger:
SQL
CREATE OR REPLACE FUNCTION assign_task_number()
RETURNS TRIGGER AS $$
DECLARE
next_num INTEGER;
BEGIN
UPDATE businesses
SET next_task_number = next_task_number + 1
WHERE id = NEW.business_id
RETURNING next_task_number - 1 INTO next_num;
NEW.task_number := next_num;
RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_assign_task_number
BEFORE INSERT ON tasks
FOR EACH ROW EXECUTE FUNCTION assign_task_number();

10.2 Remark Debounce Logic
On remarks.INSERT, Edge Function checks notifications WHERE task_id = this AND event_type = remarked AND push_sent_at > NOW() - INTERVAL 10 minutes
If recent push exists: skip push, insert notification with push_sent_at = NULL (in-app only)
If no recent push: push to ALL owner devices in push_subscriptions, insert notification
HTTP 410 Gone response: delete that push_subscriptions row
10.3 Draft Reminder Logic
Edge Function check-draft-reminders runs every 4 hours
For each business where account_status IN (trial_active, active):
Count tasks WHERE is_draft = true AND created_at < now() - interval '4 hours'
Check notifications WHERE business_id = this AND event_type = 'draft_reminder' AND push_sent_at > NOW() - INTERVAL 8 hours
If drafts exist AND no recent reminder: push to owner, insert notification
Push body: "You have X incomplete draft tasks"

11.2 Account Suspension — Security Actions
On suspension: (1) account_status = suspended on businesses; (2) Supabase Admin API signOut(userId, {scope: "global"}) for owner + all staff; (3) All push_subscriptions rows for all users in business deleted; (4) In-app notification sent to owner and all staff
Suspended users who use stale JWTs receive 401 on all Supabase operations (RLS account_status check)
On reactivation: sessions restored naturally on next login; users re-subscribe to push on next app open
11.3 Superadmin Account Security
Single Superadmin account — seeded via one-time SQL script, not self-registration
Superadmin email and initial password stored in secure password manager (not in repo or env files)
Superadmin can change their own password via Supabase Auth (/sa/settings)
MFA recommended: Enable Supabase MFA (TOTP) for the Superadmin account via Supabase dashboard. Not enforced in app code for v1.0, but documented as a setup step.
No forgot password email for Superadmin — recovery via Supabase Dashboard Auth tab directly
11.4 Additional Security Measures
Rate limiting on login: Supabase Auth provides basic rate limiting. Additionally, the login form disables the submit button for 30 seconds after 5 consecutive failed attempts (client-side).
Staff password reset: Only the Owner can reset a staff member's password via Admin Panel. This uses the Edge Function with service role, not a direct Supabase Admin API call from the frontend.
No customer PII in push bodies: Prevents data leakage on lock screens.
Service role key isolation: SUPABASE_SERVICE_ROLE_KEY is ONLY in Supabase Edge Function environment. Never in frontend code or Vercel environment.
